2. Workflow — End to End

Flow: Student registers → uploads CV → Android sends to Pyhto AI / Python backend → CV parsed (spaCy NER) → matched against jobs (BERT) → shortlisted → ranked results → apply → interview (Pyhto Q&A) → result stored in Firestore

4. Scoring Formula

Total (0–100) = Skill Overlap (60%) + BERT Cosine (10%) + Experience (15%) + Education (10%) + Extra (5%)

ComponentWeightCalculation
Skill Overlap60 pts(|matching_skills| / |job_skills|) × 60
BERT Semantic10 ptscosine_sim(CV_embed, job_embed) × 10
Penalty: if skill_ratio < 0.3 AND cos_sim > 0.3 → ×0.4
Experience15 pts≥5yr=15, ≥2yr=12, ≥1yr=8, >0=4, none=0
Education10 ptsPhD=10, Master=8, Bachelor=6, Other=4, none=0
Extra5 ptsGPA(0-1.5) + Certs(0-2) + Projects(0-1.5)

BERT Penalty Example

John (Python/Java/React) → Data Scientist job [python, ML, tensorflow, sql]
  skill_ratio = 1/4 = 0.25  →  < 0.3  ✓ trigger
  cos_sim    = 0.55         →  > 0.3  ✓ trigger
  → bert_score ×= 0.4       →  penalty applied
  Reason: low skill overlap + moderate BERT similarity → likely domain mismatch

9. Interview Preparation — Key Questions & Answers

Q: REST API kaise use kiya?
A: FastAPI framework use kiya. 4 endpoints design kiye: POST /api/parse-cv (PDF upload → structured CV), POST /api/match (CV+Job → match score), POST /api/shortlist (candidates → ranked list), POST /api/recommend (student → job recommendations). Sab endpoints Pydantic se validate hote hain aur OpenAPI docs auto-generate hoti hain. Firebase Android app direct use karta hai (SDK internally REST calls karta hai). Python backend microservice as a separate tier deploy hota hai.
Q: Models kyun select kiye?
A: spaCy en_core_web_lg — pre-trained NER system, zero training cost, 90%+ accuracy. all-MiniLM-L6-v2 — lightest BERT (80MB, 6 layers), fast inference, good for short text. n-gram fallback — zero dependencies, pure Python. Graceful degradation pattern follow kiya — agar model install nahi bhi ho to bhi system pure regex se kaam karta hai.
Q: Scoring formula kaise design ki?
A: Skill overlap ko 60% weight diya — job matching mein skills primary factor hain. BERT sirf 10% weight ke saath fine-tuning karta hai — embeddings ka limited effect ensure karta hai. Experience 15%, Education 10%, Extra (GPA+Certs+Projects) 5%. BERT penalty condition: agar skill_ratio < 0.3 aur cos_sim > 0.3 hai to BERT score ×0.4 ho jata hai — domain mismatch cases handle karta hai.
Q: Challenges kya the?
A: (1) Skill extraction mein false positives — "c" match ho raha tha "account" se, word boundaries fix se precision 68% to 87% gayi. (2) Kaggle CVs mein inconsistent formatting — date-based fallback add kiya, experience detection 45% to 95%+. (3) BERT ka limited effect — deliberately 10% weight rakha kyunki embeddings generic hain, skills specific nahi. (4) spaCy model 400MB ka hai — slow connections pe timeout hota hai, isliye automatic fallback daala.
